# Infrastructure and Deployment

The Green Moment Backend V2 uses a modern, serverless architecture entirely hosted on Google Cloud Platform (GCP).

## Infrastructure Components

1. **Compute Engine (Serverless): Google Cloud Run**
   - The FastAPI app is containerized via Docker and runs on Cloud Run.
   - **Why Cloud Run?** It scales down to zero when not in use, saving costs, but can rapidly scale up to handle sudden spikes in client traffic. It is configured to be memory-intensive (2Gi) rather than GPU-based, so inference runs purely on the CPU.
2. **Databases & Storage:**
   - **Google Cloud Storage (GCS):** Acts as the primary object store. The ML artifacts (`carbon_intensity.json`, `ml_cache_v2.json`) are stored here for fast, cheap access.
   - **Firestore:** A NoSQL document database used exclusively for managing user profiles and alert preferences.
3. **Task Orchestration: Google Cloud Scheduler**
   - Acts as a serverless cron job.
   - Triggers the internal pipelines (Data ETL/Inference and Notifications) via authenticated HTTP POST requests.
4. **Authentication & Identity:**
   - **Firebase Authentication:** Handles mobile user identities (JWTs).
   - **Firebase App Check:** Prevents unauthorized API access from malicious bots or scripts by verifying Play Integrity.
   - **Google IAM & Service Accounts:** Secures server-to-server communication (e.g., Cloud Scheduler calling Cloud Run).
5. **External Services:**
   - **Bright Data:** A proxy network used to fetch data from Taipower without getting rate-limited or IP-blocked.

## Deployment Process

Deployment is automated via the `scripts/deploy.sh` script.

### 1. Build Phase
The script uses **Google Cloud Build** to build the Docker image remotely and push it to the Google Container Registry (GCR).
```bash
gcloud builds submit --tag gcr.io/$PROJECT_ID/gm-backend-v2 --project $PROJECT_ID
```

### 2. Deployment Phase
The script deploys the newly built image to Cloud Run with the following configurations:
- `--platform managed`: Uses the fully managed Cloud Run environment.
- `--allow-unauthenticated`: Essential for allowing the mobile app to communicate with the Client endpoints (security is handled internally by Firebase App Check and Auth).
- `--service-account`: Binds the Cloud Run instance to a specific Service Account (`gm-backend-v2-identity`) that has exact permissions for Firestore, GCS, and Secret Manager.
- `--cpu 1 --memory 2Gi`: Allocates sufficient memory for loading the PyTorch ML models into memory.
- `--min-instances 0`: Allows scaling to zero.
- `--max-instances 3`: Caps scaling to control costs.
- `--concurrency 80`: Handles up to 80 concurrent requests per instance.

### 3. Post-Deployment (Cloud Scheduler)
After deployment, the `deploy.sh` script outputs the new Service URL. 
The developer must ensure that Google Cloud Scheduler is configured to hit `<URL>/api/v1/internal/update-pipeline` and `<URL>/api/v1/internal/dispatch-notifications` using an OIDC Token generated by the authorized Service Account.
